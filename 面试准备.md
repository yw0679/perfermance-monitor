# 面试准备：分布式服务器性能监控系统

---

## 一、项目整体介绍（1-3分钟口述稿）

> 我做的是一个面向高并发场景的**分布式服务器性能监控系统**，背景是学校选课/查成绩系统——这类场景 CPU 和 IO 密集，需要对服务器集群做实时健康度评估。
>
> 系统采用 **hub-and-spoke 架构**：一个中心管理节点 manager，多个被监控节点 worker。每个 worker 上跑了**六个采集模块**：
> - **CPU 百分比** → 自研内核模块通过 mmap 共享内存提供原始累计值，worker 端做差分计算
> - **CPU 负载** → 读 `/proc/loadavg` 获取 1/5/15 分钟负载
> - **内存** → 读 `/proc/meminfo` 获取 19 个字段
> - **网络** → eBPF TC hook 挂到物理网卡上做字节/包计数
> - **磁盘** → 读 `/proc/diskstats` 获取 IOPS、吞吐、利用率
> - **主机信息** → hostname + 主网卡 IPv4
>
> 采集的数据用 **protobuf 序列化**后通过 **gRPC** 每 1 秒同步推送到 manager。
>
> Manager 收到数据后做三件事：
> 1. 用**加权评分模型**（CPU 35%、内存 30%、负载 15%、磁盘 15%、网络 5%）算出每台服务器的 0-100 健康分
> 2. 把历史数据通过**异步队列**落库到 MySQL（三张表：性能总表、网络明细、磁盘明细）
> 3. 对外提供 gRPC 查询接口：查最优主机、集群概览、历史性能曲线
>
> 前端还有个 **Flask + Chart.js** 的 web dashboard，4 条实时曲线 + 4 个指标卡片，秒级刷新。
>
> 技术栈：**C++17 + gRPC + eBPF + Linux 内核模块 + MySQL + protobuf + Python Flask + Chart.js**。整个系统从内核态到用户态、从数据采集到可视化，全链路独立完成。

## 二、技术深挖问答

### Q1: 为什么 CPU 采集用内核模块而不是直接读 `/proc/stat`？

**答：** `/proc/stat` 按行输出文本，需要字符串解析、内存拷贝和系统调用开销。每秒一次问题不大，但如果未来要提升采样频率（比如 100ms），开销就很明显了。

内核模块的做法是：
- **高精度定时器**（`hrtimer`）直接在**内核上下文中**调用 `kcpustat_cpu_fetch()` 采集原始 CPU 累计时间，写入 `vmalloc_user()` 分配的共享内存
- 用户态通过 `mmap()` 映射 `/dev/cpu_stat_monitor` 做**零拷贝**读取
- 中间用**序号锁（seq-lock）**保证读到的是原子一致的快照——写者更新前后各改一次 `seq`（奇数→偶数），读者先读 `seq` → 本地 `memcpy` 一份快照 → 再读 `seq`，确认前后一致且为偶数则数据有效，否则重试（最多 5 次）

这适合"一写多读"场景，避免了锁竞争。

**关键代码：**
- 写者：`kernel/cpu_stat_module/cpu_stat_collector.c:89-113`（`collect_all_cpu_stats`）
- 读者：`worker/src/monitor/cpu_stat_monitor.cpp:176-207`（`CopyMappedCpuStats`）

### Q2: 序号锁（seq-lock）具体是怎么实现的？

**答：**

**写者（内核模块）：**
```c
// seq 从偶数变为奇数，表示"开始写入"
WRITE_ONCE(g_shared_region->seq, seq_before + 1);
smp_wmb();  // 写屏障：确保后续写入在 seq 变更之后对读者可见

// 写入数据
collect_cpu_stat_raw(cpu, &g_shared_region->cpus[cpu]);

smp_wmb();
// seq 从奇数变为偶数，表示"写入完成"
WRITE_ONCE(g_shared_region->seq, seq_before + 2);
```

**读者（用户态 Worker）：**
```cpp
for (int attempt = 0; attempt < 5; ++attempt) {
    uint32_t seq_before = region->seq;
    if ((seq_before & 1) != 0) {  // 奇数 = 正在写入，等
        usleep(1000);
        continue;
    }
    
    cpu_stat_shared_region snapshot;
    std::memcpy(&snapshot, region, sizeof(snapshot));  // 本地拷贝
    
    uint32_t seq_after = region->seq;
    if (seq_before != seq_after || (seq_after & 1) != 0) {
        // 写入中断或 seq 变了，重试
        usleep(1000);
        continue;
    }
    
    return BuildCpuStatsFromSnapshot(snapshot, stats);  // 数据一致
}
```

**关键点：** 用 `READ_ONCE`/`WRITE_ONCE` 防止编译器优化导致的乱序，用 `smp_wmb()` 内存屏障保证多核可见性。

### Q3: CPU 百分比是怎么从累计值算出来的？

**答：** 内核模块只提供**绝对累计时间**（user、system、idle 等的总微秒数），本身没有百分比。

Worker 端维护了一个 `cpu_stat_map_` 缓存上一轮的值：

```cpp
// cpu_stat_monitor.cpp:246-253
uint64_t total_delta = current_total - previous_total;
uint64_t busy_delta = current_busy - previous_busy;

float cpu_percent = (busy_delta * 100.0) / total_delta;

// 细分百分比同理
float usr_percent = (delta_user * 100.0) / total_delta;
float idle_percent = (delta_idle * 100.0) / total_delta;
```

其中 `BusyTime = user + nice + system + irq + softirq + steal`（不包含 idle 和 iowait）。

**边界处理：**
- 第一次采样无上一轮数据 → CPU% = 0
- `total_delta == 0` 或新值小于旧值 → 返回 0
- 代码见 `cpu_stat_monitor.cpp:99-106`（`SafePercent` 函数）

### Q4: eBPF TC hook 是怎么工作的？为什么选 TC 而不是 XDP？

**答：**

**选型理由：** XDP 只能抓入方向（ingress），而我们需要**同时统计收发两个方向**。TC（Traffic Control）原生支持 `BPF_TC_INGRESS` + `BPF_TC_EGRESS`，位于网络协议栈 L2/L3 边界，能访问完整 `skb` 结构。

**工作流程：**

1. **编译加载**（`worker/src/ebpf/net_stats.bpf.c`）：
   - Clang 编译 BPF C → 字节码
   - `bpftool gen skeleton` 生成 C header（`net_stats.skel.h`）
   - 用户态调用 `net_stats_bpf__open()` → `__load()` → 获取 `map_fd` 和 `prog_fd`

2. **挂载到网卡**（`worker/src/monitor/net_ebpf_monitor.cpp:190-210`）：
   ```cpp
   for (ifindex : GetAllIfIndexes()) {
       // 创建 clsact qdisc
       bpf_tc_hook_create(&hook);
       // 先 detach 旧程序，再 attach
       bpf_tc_detach(&hook, &detach_opts);
       bpf_tc_attach(&hook, &attach_opts);  // ingress + egress 各一次
   }
   ```
   同一个加载好的 BPF 程序复用到所有物理网卡上，通过 `skb->ifindex` 做 map key 区分。

3. **内核态数据路径**（`net_stats.bpf.c:93-107`）：
   ```c
   SEC("tc/ingress")
   int tc_ingress(struct __sk_buff *skb) {
       update_stats(skb->ifindex, skb->len, /*is_rx=*/true);
       return TC_ACT_OK;  // 不影响数据包正常处理
   }
   ```

4. **原子更新**（`net_stats.bpf.c:60-86`）：
   ```c
   __atomic_fetch_add(&stats->rcv_bytes, len, __ATOMIC_SEQ_CST);
   __atomic_fetch_add(&stats->rcv_packets, 1, __ATOMIC_SEQ_CST);
   ```
   用 `__ATOMIC_SEQ_CST` 保证多 CPU 并发安全。

5. **用户态读 map + 算速率**（`net_ebpf_monitor.cpp:259-321`）：
   - `bpf_map_get_next_key()` 遍历所有 ifindex
   - `bpf_map_lookup_elem()` 读累计值
   - 和缓存值差分：`rate = delta_bytes / 1024.0 / elapsed_seconds`
   - 溢出保护：新值 < 旧值时直接用新值

**为什么用 HASH Map 而不是 ARRAY？**
- ifindex 不连续（如 1, 2, 3, 1000），HASH 更省空间
- 多网卡需要按 key 独立维护状态

### Q5: 怎么判断是物理网卡而不是虚拟网卡/lo？

**答：**

```cpp
// net_ebpf_monitor.cpp:37-46
bool IsPhysicalInterface(const std::string& ifname) {
    if (ifname == "lo") return false;  // 硬编码排除
    
    // 物理网卡（以及 virtio/pci 网卡）在 sysfs 下有 device 符号链接
    // 虚拟网卡（docker/veth/bridge）没有
    const std::string device_path = "/sys/class/net/" + ifname + "/device";
    struct stat st = {};
    return lstat(device_path.c_str(), &st) == 0;
}
```

这是 **难点之一**：如果不区分，TC hook 挂到 docker 网桥上会导致数据不准，而 `lo` 会让本地 `iperf3` 产生虚假的"网络流量"。

### Q6: Manager 怎么处理多个 worker 并发上报的线程安全问题？

**答：**

gRPC C++ 的同步 server runtime 内部有**线程池**，多个 worker 的 `SetMonitorInfo` RPC 会**并发执行**在 `GrpcServerImpl::SetMonitorInfo()`。

所有共享状态都加了**互斥锁**保护：

```
RPC Handler 线程1 ─┐
RPC Handler 线程2 ─┼─→ GrpcServerImpl::SetMonitorInfo()
RPC Handler 线程3 ─┘       │
                            ├─ lock → 写 host_data_（存最新 MonitorInfo 原文）
                            ├─ lock → 写 host_scores_（评分 + 调度状态）
                            ├─ callback → HostManager::OnDataReceived()
                            │          ├─ 评分计算（无锁，每线程独立计算）
                            │          ├─ 变化率计算（读 last_perf_samples，用 mutex）
                            │          ├─ 更新 host_scores_（lock）
                            │          └─ 异步入队 MySQL 写任务（lock + cv.notify_one）
                            └─ 返回 Status::OK
                                    ↓
MySQL 写线程：               wait(cv) → lock → pop 任务 → 长连接写库
```

**关键设计决策：**
- MySQL 写入是**异步**的——RPC handler 只把任务入队就返回 OK，不等磁盘 IO
- 后台单独一条 `MysqlWriteLoop` 线程消费队列，用 `condition_variable` 驱动
- gRPC 返回成功 ≠ MySQL 已落盘，只表示 manager 已接收并完成内存态处理

### Q7: 评分模型怎么设计的？为什么权重这样分配？

**答：**

面向选课/查成绩这种 **CPU 密集 + IO 密集** 场景设计：

| 维度 | 权重 | 设计逻辑 |
|------|------|----------|
| CPU 使用率 | 35% | 选课系统大量并发计算，CPU 是直接瓶颈 |
| 内存使用率 | 30% | 内存不够会导致频繁换页，严重拖慢性能 |
| CPU 负载 | 15% | 负载反映排队程度，IO 密集型允许一定过载 |
| 磁盘 IO 利用率 | 15% | 数据库读写密集，磁盘利用率直接影响响应时间 |
| 网络带宽 | 5%（收发各 2.5%） | 选课场景不是网络瓶颈型，收发分离体现独立状态 |

**计算过程（`host_manager.cpp:446-510`）：**

```cpp
// 1. 提取各维度数据
cpu_percent   // 总 CPU 百分比
mem_percent   // 已使用内存百分比
load_avg_1    // 1 分钟负载
disk_util     // 磁盘利用率（取所有磁盘最大值）
net_recv_rate // 接收字节速率
net_send_rate // 发送字节速率

// 2. 反向归一化：使用率越低分数越高
cpu_score  = clamp(1.0 - cpu_percent / 100.0)
mem_score  = clamp(1.0 - mem_percent / 100.0)
load_score = clamp(1.0 - load_avg_1 / (cpu_cores * 1.5))  // IO 密集系数 1.5
disk_score = clamp(1.0 - disk_util / 100.0)
net_recv_score  = clamp(1.0 - net_recv_rate / 125000000.0)  // 1Gbps
net_send_score  = clamp(1.0 - net_send_rate / 125000000.0)
net_score = (net_recv_score + net_send_score) / 2.0

// 3. 加权求和 → 0-100
score = (cpu_score * 0.35 + mem_score * 0.30 + load_score * 0.15
         + disk_score * 0.15 + net_score * 0.05) * 100.0

score = clamp(score, 0, 100)
```

**负载系数 1.5 的含义：** I/O 密集场景下允许 CPU 负载适度超过核心数（等待 IO 不算真正的 CPU 压力），所以归一化分母从 `cpu_cores` 放大到 `cpu_cores × 1.5`。

### Q8: 为什么用 protobuf 而不是 JSON？

**答：**

1. **强类型 + Schema 契约**：proto 文件定义了 worker 和 manager 之间严格的数据契约，字段变更时编译器会直接报错
2. **二进制序列化更紧凑**：比 JSON 小 3-10 倍，1 秒一次高频上报省带宽
3. **gRPC 天然配套**：定义一次 `.proto` 文件就同时生成客户端 stub 和服务端基类

`proto/monitor_info.proto` 定义了 worker → manager 的上报格式：

```protobuf
message MonitorInfo {
    string name = 1;
    HostInfo host_info = 2;
    repeated CpuStat cpu_stat = 3;       // per-core CPU 数据
    CpuLoad cpu_load = 4;
    MemInfo mem_info = 5;
    repeated NetInfo net_info = 6;       // per-网卡数据
    repeated DiskInfo disk_info = 7;     // per-磁盘数据
}

service GrpcManager {
    rpc SetMonitorInfo(MonitorInfo) returns (google.protobuf.Empty);
}
```

### Q9: gRPC worker 调用 SetMonitorInfo 会阻塞吗？怎么处理的？

**答：** 会阻塞。当前是**同步 unary RPC**：

```cpp
// monitor_pusher.cpp:94-113
grpc::Status status = stub_->SetMonitorInfo(&context, info, &response);
// ↑ 当前线程会阻塞等待 RPC 完成，或被超时/取消打断
```

**超时保护：** 设置了 3 秒 deadline（行 101-102）：
```cpp
context.set_deadline(std::chrono::system_clock::now() + std::chrono::seconds(3));
```
超时后 gRPC 框架自动取消调用，线程返回 `DEADLINE_EXCEEDED`。

**失败处理：** 当前是记日志然后下一轮再发（1秒间隔），属于"尽力而为推送"模式。`grpc_discussion_summary.md` 中讨论了更完善的方案：有限重试 + 退避抖动 + 熔断降频。

### Q10: Manager 端 gRPC server 如何具备并发能力？

**答：** Manager 启动后不只是一个普通 C++ 程序，而是运行在 **gRPC server runtime** 之上。gRPC 框架提供了：
- HTTP/2 多路复用
- 内部线程池调度
- 请求分发路由

类比 Web 开发：我们写 Controller，Tomcat/Netty 负责监听端口和并发调度。不需要手写 socket 监听或线程池，但共享状态的并发安全需要自己加锁。

**代码入口：** `manager/src/main.cpp`
```cpp
GrpcServerImpl service;
grpc::ServerBuilder builder;
builder.AddListeningPort(address, grpc::InsecureServerCredentials());
builder.RegisterService(&service);                      // 注册服务实现
std::unique_ptr<grpc::Server> server(builder.BuildAndStart());
// ...
server->Wait();  // 进入服务等待
```

### Q11: 变化率是怎么计算的？为什么需要变化率？

**答：** 评分只代表"当前状态"，但快速恶化比静态高分更值得关注。所以 manager 在每次收到数据后计算**变化率**：

```cpp
// host_manager.cpp（变化率计算逻辑在 OnDataReceived 中）
cpu_percent_rate     = (current.cpu - last.cpu) / elapsed_seconds
mem_used_rate        = (current.mem - last.mem) / elapsed_seconds
load_rate            = (current.load - last.load) / elapsed_seconds
disk_util_rate       = (current.disk - last.disk) / elapsed_seconds
net_in_rate          = (current.net_in - last.net_in) / elapsed_seconds
net_out_rate         = (current.net_out - last.net_out) / elapsed_seconds
```

这些变化率字段随评分一起写入 MySQL `server_performance` 表，可以用于**趋势告警**：即使分数不低，但 CPU 使用率 5 秒内飙升 50%，说明有问题。

### Q12: 为什么 MySQL 写入是异步队列而不是同步？

**答：**

1. **gRPC RPC 不应该被磁盘 IO 拖慢**：如果 `SetMonitorInfo` 里直接执行 `mysql_query(INSERT)`，磁盘抖动会导致所有 worker 的 RPC 响应延迟飙升
2. **解耦保证吞吐**：入队只要微秒级，写入可以攒一批一起做
3. **连接复用**：后台线程持有一个 MySQL 长连接，避免每次 INSERT 都建连/断连

具体实现（`host_manager.cpp:514-648`）：
- `std::deque<MysqlWriteTask>` 做缓冲
- `std::mutex + condition_variable` 做线程同步
- 后台 `MysqlWriteLoop` 线程阻塞在 `cv.wait` 上，有新任务就 pop + 写入

### Q13: 内存屏障（`smp_wmb`）在代码中的作用？

**答：** 在 `cpu_stat_collector.c:102,111` 中：

```c
WRITE_ONCE(g_shared_region->seq, seq_before + 1);
smp_wmb();  // ← 写屏障
g_shared_region->version = CPU_STAT_SHARED_VERSION;
g_shared_region->cpu_count = cpu_count;
// ... 写入各 CPU 数据 ...
smp_wmb();  // ← 写屏障
WRITE_ONCE(g_shared_region->seq, seq_before + 2);
```

**作用：** 保证两道 `smp_wmb()` 之间（以及其后的最终 `seq` 更新之前）的所有写入，对其他 CPU 一定是按顺序可见的。这样读者看到 `seq` 是偶数时，中间的 `version`、`cpu_count`、各 CPU 数据一定都已写入完毕。

### 如果监控 10000 台服务器，当前架构有什么瓶颈？

| 瓶颈点 | 问题分析 | 解决方案 |
|--------|----------|----------|
| **Manager 单点容量** | 单进程内存和 CPU 扛不住 10000 路并发 gRPC | 1) Manager 做无状态分片，前面放 Envoy/L4 LB 按 host hash 路由<br>2) 分层架构：worker → 区域 collector → 全局 aggregator |
| **MySQL 写入** | 10000 QPS INSERT，单表很快膨胀 | 1) 批量写（攒 100 条一批 commit）<br>2) 分区表（按天/小时）<br>3) 考虑时序数据库：TDengine / ClickHouse / TimescaleDB |
| **网络带宽** | 每 worker 单次上报约 2KB，10000 台 = 20 MB/s | 还好，1Gbps 链路不是瓶颈 |
| **内存** | `host_scores_` 存 10000 份评分可接受，但 `host_data_` 存完整 `MonitorInfo` proto 会很大 | 只保留评分需要的摘要字段，原文只在写入队列中短驻 |

### Push vs Pull 模型怎么选？

**当前是 Push 模型：**
- 优势：实时性好，manager 不需要知道所有 worker 的地址，worker 部署灵活
- 劣势：manager 被动接收，无法控制采集节奏；worker 故障只能靠超时判活（60s 无数据 = 离线）

**生产系统建议 Push + Pull 混合：**
- 正常情况：Worker Push 上报指标数据
- 健康检查：Manager 定期 Pull 每个 worker 的心跳（轻量 RPC）
- 配置下发：Manager 可以通过 RPC 动态调整 worker 的采集频率、具体采集哪些指标

### 如何扩展成可告警的监控系统？

1. **规则引擎**：在 `HostManager::CalcScore()` 之后增加规则评估，如 `score < 30 → CRITICAL`
2. **告警通道**：新增告警队列，异步推送（钉钉/邮件/企业微信）
3. **告警抑制**：连续 N 次触发才告警，避免抖动
4. **变化率告警**：利用已有的 `_rate` 字段做突增告警（如 CPU 5 秒涨 30%）

---

### 如何监控"监控系统"本身？

- **Manager 自身指标**：收到请求数、处理延迟 p50/p99、MySQL 队列深度、内存使用
- **Worker 自身指标**：推送成功率、连续失败次数、采集耗时
- **自举（Bootstrap）**：Manager 可以把自身也当作一个 Worker，向自己推送指标？但要注意循环依赖问题（Manager 挂了谁帮它上报）
- **更好的方案**：独立部署一个健康检查服务，只负责探测 Manager 和 Worker 的存活

## 四、项目难点 & 解决方案

### 难点 1：内核模块共享内存的一致性

**问题：** 内核模块通过 `hrtimer` 每秒更新共享内存，用户态 `mmap` 后直接读。如果读到一半内核又更新了，会得到半新半旧的不一致数据。

**解决：** 实现序号锁（seq-lock），见 Q2 详细说明。读者不做锁，只做"读 seq → memcpy 本地快照 → 再读 seq"的重试循环。一次写入、多次读取场景最优解。

**收益：** 零拷贝 + 无锁读，适合高并发读取场景。

---

### 难点 2：eBPF 程序只监控物理网卡

**问题：** Docker、veth、bridge 等虚拟网卡在 sysfs 下有 `/sys/class/net/X/device` 但行为不同，TC hook 挂上去数据不准；`lo` 回环接口会统计本地流量，产生虚假数据。

**解决：** 
- 硬编码排除 `lo`
- `lstat("/sys/class/net/<ifname>/device")` 判断是否为真物理设备——只有物理网卡和 virtio 这类真实硬件才有 `device` 符号链接
- 代码：`worker/src/monitor/net_ebpf_monitor.cpp:37-46`

---

### 难点 3：变化率计算的边界条件

**问题：** 各种边界情况需要防御性处理。

| 场景 | 处理方式 | 代码位置 |
|------|----------|----------|
| 第一次采样无历史数据 | CPU% = 0，网络速率 = 0（跳过不填充） | `cpu_stat_monitor.cpp:290-291`, `net_ebpf_monitor.cpp:287-288` |
| 累积值回退（理论上不应发生） | 直接用新值作为 delta | `cpu_stat_monitor.cpp:100-101`, `net_ebpf_monitor.cpp:293-297` |
| 时间间隔 ≤ 0 | 跳过不计算速率 | `net_ebpf_monitor.cpp:291-292` |
| 除数为 0 | 返回 0 | `cpu_stat_monitor.cpp:99-101` |

---

### 难点 4：gRPC 超时与 worker 线程保护

**问题：** 初始版本没设 deadline，如果 manager 挂了，worker 推送线程会一直阻塞在 `stub_->SetMonitorInfo()`，永不返回。

**解决：** 加了 3 秒 deadline（`monitor_pusher.cpp:101-102`）：
```cpp
context.set_deadline(std::chrono::system_clock::now() + std::chrono::seconds(3));
```
超时后 gRPC 框架自动取消调用，线程返回 `DEADLINE_EXCEEDED`。

**更进一步的设计讨论（见 `grpc_discussion_summary.md`）：**
- 有限重试 + 指数退避
- 熔断降频：连续失败达到阈值后拉长上报间隔
- 幂等保护：配合重试需要防止重复写入

---

### 难点 5：跨内核模块和 eBPF 的编译依赖链

**问题：** 项目编译依赖复杂：
- C++ 代码 → CMake（protobuf 生成、gRPC 生成、链接）
- 内核模块 → Make（需要 linux-headers，用内核构建系统）
- eBPF → Make（需要 clang + bpftool + libbpf，生成 skeleton header）

**解决：** CMakeLists.txt 中用 `add_custom_command` 串联：
```
CMake build → proto/ 编译 → 静态库 libproto.a
           → worker/ 链接 libproto + libbpf + grpc
           → manager/ 链接 libproto + grpc + mysqlclient
           → [触发] make -C worker/src/ebpf  # 生成 net_stats.skel.h
```
独立编译项（内核模块、前端）通过 make 和 shell 脚本管理。

## 五、Mock 模拟面试

### 面试官：介绍一下你简历上的这个监控项目？

**建议回答 →** 用第一部分的 1-3 分钟口述稿。

---

### 面试官：为什么不用 Prometheus + Node Exporter，要自己写？

**建议回答：**

> 首先，业务场景要求**秒级实时推送**和**自定义评分模型**——Prometheus 默认是 Pull 模式 15s 拉一次，秒级采集需要调配置，而 Push 模式对管理节点不知道 worker 地址的场景更灵活。
>
> 第二，我需要一个**综合评分模型**来量化"服务器适不适合接收更多请求"，这不是 Prometheus 原生能力。
>
> 第三，这是一个**学习项目**，我想通过它深入理解内核模块、eBPF、gRPC 的完整工程链路。用现成工具虽然快，但学不到"为什么 CPU 百分比要差分计算""序号锁怎么保证并发一致性"这些底层知识。
>
> 如果是真实生产环境且团队有运维能力，我会优先推荐 Prometheus + Grafana + Alertmanager 这种成熟方案。

---

### 面试官：内核模块里用 `hrtimer` 每秒一次，为什么不直接读 `/proc/stat`？

**建议回答 →** 见 Q1。

---

### 面试官：如果 eBPF 程序加载失败，worker 还能工作吗？

**建议回答：**

> 能。采集模块之间是**解耦**的——每个模块实现 `MonitorInter` 接口，`MetricCollector::CollectAll()` 依次调用每个模块的 `UpdateOnce()`。eBPF 加载失败时 `NetEbpfMonitor` 内部标记 `loaded_ = false`，后续 `UpdateOnce` 直接 return，不影响其他模块继续采集 CPU、内存、磁盘。
>
> 不过 worker 启动时需要 `sudo` 权限，否则 eBPF 程序无法 attach TC hook 到网卡上。

---

### 面试官：你提到 gRPC 是同步阻塞调用，如果 manager 挂了会怎样？

**建议回答：**

> Worker 推送线程在 `stub_->SetMonitorInfo()` 上阻塞，但有 3 秒 deadline 保护。超时后返回 `DEADLINE_EXCEEDED`，打日志，然后间隔 1 秒继续下一轮尝试。
>
> 如果是生产系统，我还会加三个优化（已经在设计文档里讨论了）：1）有限重试 + 指数退避；2）熔断机制——连续失败 10 次就把推送间隔从 1 秒拉长到 10 秒，60 秒后逐步恢复；3）幂等写入——避免重试导致 MySQL 重复记录。

---

### 面试官：如果要支持 Windows 监控，你会怎么改？

**建议回答：**

> 核心改动：内核模块和 eBPF 是 Linux 独占的，需要做平台抽象。

> 1. **CPU 采集**：Linux 用内核模块 + mmap，Windows 用 Windows API（`GetSystemTimes`、`NtQuerySystemInformation`）
> 2. **网络采集**：Linux 用 eBPF TC，Windows 用 Win32 API（`GetIfTable2` / `GetPerTcpConnectionEStats` 或 ETW Trace）
> 3. **采集接口不变**：`MonitorInter` 接口是 C++ 虚类，可以用 `#ifdef` 或编译期多态替换实现
> 4. **gRPC + protobuf**：跨平台的，无需改动
>
> 工作量预估：CPU 和内存采集 1-2 天，网络是最大难点（Windows 没有 eBPF 等价物，需要研究 Event Tracing for Windows），可能 3-5 天。总体 2 周内可完成。

---

### 面试官：系统里有没有考虑优雅退出？

**建议回答：**

> 有。`MonitorPusher::Stop()` 设置 `running_ = false`，推送循环退出，然后 `thread_.join()` 等待线程结束。`HostManager::Stop()` 也是一样：`running_ = false` + `mysql_queue_cv_.notify_all()` 唤醒阻塞的 MySQL 写线程 + `join()`。
>
> eBPF 程序的清理也在析构时做：`CleanupEbpf()` 逐网卡 detach TC hook + destroy qdisc，最后 `net_stats_bpf__destroy(skel_)` 释放资源。

---

### 面试官：如果让你从头再做这个项目，有什么想改的？

**建议回答：**

> 1. **加单元测试**：当前只有手动压测验证，没有自动化测试。Score 计算、CPU% 计算、SafePercent 这些纯函数非常适合写单元测试
> 2. **配置化**：MySQL 连接信息、评分权重、负载系数都硬编码了，应该读配置文件
> 3. **结构化日志**：当前 `std::cout`/`std::cerr`，应该用 spdlog 或 glog，加上日志级别和滚动
> 4. **健康检查接口**：Manager 和 Worker 各加一个 Health RPC，方便外面做存活探测
> 5. **数据持久化方案**：MySQL 对时序数据不是最优，如果重来会选项 TDengine 或 InfluxDB
> 6. **容器化部署**：写 Dockerfile + docker-compose，一键拉起 manager + worker + MySQL + 前端，降低部署门槛
> 7. **Worker 侧加本地缓存**：网络断连时不丢数据，恢复后补推历史

## 六、核心数据流速查

```
采样触发 (2条路径)
├── 内核模块 hrtimer → 每秒写共享内存 → /dev/cpu_stat_monitor
└── eBPF TC hook → 每个数据包触发 → BPF HASH map (ifindex → net_stats)

Worker 推送线程 (1秒周期)
├── MetricCollector::CollectAll()
│   ├── CpuStatMonitor  → open/read/mmap→差分→FillCpuPercentages
│   ├── CpuLoadMonitor  → /proc/loadavg
│   ├── MemMonitor      → /proc/meminfo
│   ├── NetEbpfMonitor  → bpf_map_get_next_key/lookup→差分→算速率
│   ├── DiskMonitor     → /proc/diskstats
│   └── HostInfoMonitor → gethostname/getifaddrs
├── Pack → MonitorInfo protobuf
├── gRPC SetMonitorInfo(3s deadline) → Manager

Manager 接收
├── GrpcServerImpl::SetMonitorInfo (并发 handler)
├── HostManager::OnDataReceived
│   ├── CalcScore (加权评分, 0-100)
│   ├── CalcRates (变化率: cpu_rate, mem_rate, ...)
│   ├── UpdateHostScores_ (内存态, mutex)
│   ├── UpdateScheduleStates (加权轮询)
│   └── Enqueue mqsql_queue_ (异步, mutex + cv)
└── MysqlWriteLoop 后台线程
    ├── INSERT server_performance
    ├── INSERT server_net_detail
    └── INSERT server_disk_detail

前端查询
├── Flask API (port 8000) → MySQL → JSON
└── Browser (port 8080) → Chart.js → 4条曲线 + 4个卡片
```
