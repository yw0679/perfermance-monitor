# 文件分层说明

## 1. 这份文档是干什么的

这份文档主要解决两个问题：

- 帮你快速区分仓库里哪些文件属于**当前简化版主线**
- 帮你识别哪些文件属于**eBPF 模块**、**内核模块**、**历史/预留代码**
- 帮你理清项目在运行简化版主线时，`worker -> manager -> MySQL -> query` 这条链路到底是怎么走的

为了方便你后面继续读源码、准备面试、做二次修改，下面会先给出分类规则，再按目录列总览表，最后单独解释“简化版主线运行流程”。

---

## 2. 分类规则

### 2.1 当前版本使用文件

指当前简化版主线真正会参与：

- 默认构建
- 默认运行
- 本地联调
- MySQL 落库
- 查询服务

这类文件是你现阶段最应该优先读懂、优先掌握的部分。

### 2.2 eBPF 模块文件

指与 `ENABLE_EBPF` 可选网络采集链路相关的文件。  
当前仓库保留了这部分能力，但**默认未启用**，只有打开 CMake 选项并满足依赖后才会参与构建。

### 2.3 内核模块相关文件

指与 `kmod`、字符设备、`cpu_softirq` 链路相关的文件。  
当前简化版主线不依赖这些文件即可跑通，因此它们属于**可选扩展能力**。

### 2.4 历史/预留文件

指仓库里保留着、但当前默认构建与运行流程不使用的文件。  
它们可能来自旧方案、备用实现、测试程序，或者后续准备启用的预留代码。

## 5. 简化版主线运行时的项目流程

下面这部分是“项目真实跑起来时”的调用链，也是你后面最应该会讲的部分。

## 5.1 先启动 Manager

启动命令示例：

```bash
./build/manager/manager 0.0.0.0:50051
```

Manager 启动后做的事情：

1. 进入 `manager/src/main.cpp`
2. 创建 `GrpcServerImpl`，用于接收 worker 推送的数据
3. 创建 `HostManager`，负责主机状态管理、评分、变化率计算、MySQL 写入
4. 创建 `QueryManager`，初始化 MySQL 连接
5. 创建 `QueryServiceImpl`，把查询逻辑暴露成 gRPC 查询服务
6. 用 `grpc::ServerBuilder` 同时注册：
   - 数据接收服务 `GrpcServerImpl`
   - 查询服务 `QueryServiceImpl`
7. 开始监听端口，等待 worker 上报数据

对应核心文件：

- `manager/src/main.cpp`
- `manager/src/rpc/grpc_server.cpp`
- `manager/src/host_manager.cpp`
- `manager/src/query_manager.cpp`
- `manager/src/rpc/query_service.cpp`

## 5.2 再启动 Worker

启动命令示例：

```bash
./build/worker/worker 127.0.0.1:50051 5
```

这里的 `5` 表示每 `5` 秒推送一次。  
如果不传第二个参数，默认值是 `10` 秒。

Worker 启动后做的事情：

1. 进入 `worker/src/main.cpp`
2. 解析 `manager_address` 和 `interval_seconds`
3. 创建 `MonitorPusher`
4. `MonitorPusher::Start()` 启动一个后台线程
5. 后台线程周期性执行 `PushLoop()`

对应核心文件：

- `worker/src/main.cpp`
- `worker/src/rpc/monitor_pusher.cpp`

## 5.3 Worker 周期采集本机指标

每一轮推送时，`MonitorPusher::PushOnce()` 会先构造一个 `MonitorInfo`，然后调用：

```text
MetricCollector::CollectAll()
```

`MetricCollector` 在构造函数里会注册当前主线需要的监控器：

- `CpuLoadMonitor`
- `CpuStatMonitor`
- `MemMonitor`
- `NetMonitor`（默认）
- `DiskMonitor`
- `HostInfoMonitor`

如果打开 `ENABLE_EBPF`，才会把 `NetMonitor` 换成 `NetEbpfMonitor`。

### 这一轮采样里每个监控器做什么

#### `CpuLoadMonitor`

- 读取 `/proc/loadavg`
- 填充 `1min / 5min / 15min` 负载

#### `CpuStatMonitor`

- 读取 `/proc/stat`
- 保存上一次采样值
- 用两次采样差分计算：
  - 总 CPU 使用率
  - user/system/nice/idle/iowait/irq/softirq 占比

#### `MemMonitor`

- 读取 `/proc/meminfo`
- 采集：
  - total
  - free
  - avail
  - buffers
  - cached
  - active / inactive
  - dirty / writeback
  - 等内存字段

#### `NetMonitor`

- 读取 `/proc/net/dev`
- 记录两次采样时间差
- 计算网卡收发速率和包速率
- 当前已经过滤：
  - `lo`
  - `docker*`
  - `veth*`
  - `br-*`
  - `virbr*`

#### `DiskMonitor`

- 读取 `/proc/diskstats`
- 记录两次采样差分
- 计算：
  - 读写吞吐
  - IOPS
  - 平均时延
  - 利用率
- 当前会跳过：
  - `loop*`
  - `ram*`
  - `fd*`
  - `sr*`

#### `HostInfoMonitor`

- 获取主机名
- 获取主网卡 IP
- 用于 Manager 侧做主机唯一标识

对应核心文件：

- `worker/src/monitor/metric_collector.cpp`
- `worker/src/monitor/cpu_load_monitor.cpp`
- `worker/src/monitor/cpu_stat_monitor.cpp`
- `worker/src/monitor/mem_monitor.cpp`
- `worker/src/monitor/net_monitor.cpp`
- `worker/src/monitor/disk_monitor.cpp`
- `worker/src/monitor/host_info_monitor.cpp`

## 5.4 Worker 把采集结果封装成 Protobuf 并通过 gRPC 上报

所有监控器最终都会把结果写进同一个消息对象：

```text
monitor::proto::MonitorInfo
```

这个消息定义在：

- `proto/monitor_info.proto`

它会聚合：

- 主机信息
- CPU 负载
- CPU 统计
- 内存信息
- 网络信息
- 磁盘信息

然后 `MonitorPusher` 调用：

```text
GrpcManager::SetMonitorInfo()
```

把整份监控数据发给 Manager。

对应核心文件：

- `proto/monitor_info.proto`
- `worker/src/rpc/monitor_pusher.cpp`

## 5.5 Manager 接收数据并触发回调

Manager 侧 gRPC 服务在：

- `manager/src/rpc/grpc_server.cpp`

当 `SetMonitorInfo()` 被调用时，会做三件事：

1. 从请求里取出主机名
2. 暂存到 `host_data_`
3. 调用在 `main.cpp` 中注册好的回调：

```text
HostManager::OnDataReceived()
```

这一步是整个 Manager 主线处理的核心入口。

## 5.6 HostManager 处理数据、计算评分、打印结果、写入 MySQL

`HostManager::OnDataReceived()` 会做以下事情：

1. 取主机名和 IP，拼接出主机唯一标识
2. 读取 CPU、内存、网络、磁盘等关键指标
3. 结合上一次采样结果计算变化率
4. 调用 `CalcScore()` 计算综合评分
5. 更新内存中的主机状态表 `host_scores_`
6. 调用 `WriteToMysql()` 写库
7. 在控制台打印这次接收的数据摘要

### 当前评分逻辑

当前评分模型大致基于以下维度：

- CPU 使用率
- 内存使用率
- 1 分钟负载
- 磁盘利用率
- 网络收发速率

通过归一化和加权后得到一个 `0~100` 的综合评分。

对应核心文件：

- `manager/src/host_manager.cpp`
- `manager/include/host_manager.h`

## 5.7 MySQL 持久化

当前主线会把数据写入几张核心表：

### 主表

- `server_performance`

存：

- CPU 核心汇总指标
- 负载
- 内存汇总
- 最大磁盘利用率
- 网络收发速率
- 综合评分
- 变化率
- 时间戳

### 明细表

- `server_net_detail`
- `server_disk_detail`
- `server_mem_detail`
- `server_softirq_detail`

虽然当前简化版主线不重点展示 SoftIRQ，但表结构仍然保留了扩展位。

表结构定义在：

- `manager/sql/init_server_performance.sql`

## 5.8 查询服务

Manager 启动时还会注册 `QueryServiceImpl`，它依赖 `QueryManager` 从 MySQL 查询数据。

当前已经具备的查询能力包括：

- 性能记录分页查询
- 趋势聚合查询
- 异常查询
- 评分排序查询
- 最新评分汇总
- 网络明细查询
- 磁盘明细查询
- 内存明细查询
- SoftIRQ 明细查询

查询服务链路是：

```text
QueryServiceImpl -> QueryManager -> MySQL
```

对应核心文件：

- `manager/src/rpc/query_service.cpp`
- `manager/src/query_manager.cpp`
- `proto/query_api.proto`

## 6. 简化版主线的整体调用链

你可以把它记成下面这条链：

```text
worker/src/main.cpp
  -> MonitorPusher
  -> MetricCollector
  -> CpuLoad / CpuStat / Mem / Net / Disk / HostInfo
  -> MonitorInfo(Protobuf)
  -> gRPC SetMonitorInfo
  -> manager/src/rpc/grpc_server.cpp
  -> HostManager::OnDataReceived()
  -> CalcScore()
  -> WriteToMysql()
  -> MySQL
  -> QueryService / QueryManager
```

如果面试官问你“这个项目的数据是怎么流动的”，你就按这条链去讲，基本不会乱。
