# 简化监控项目简历描述

> 使用前建议根据你的实际完成情况微调，尤其是压测结果、部署方式和查询能力等细节。

## 项目名称

基于 C++/gRPC 的 Linux 系统性能监控平台（简化版）

## 技术栈

C++17、Linux、`/proc`、gRPC、Protobuf、MySQL、CMake、Docker、stress、iperf3

## 项目描述

面向 Linux 服务器或边缘设备运行过程中“资源状态不可见、故障定位效率低”的问题，设计并实现了一套轻量级系统性能监控平台。项目采用 `Worker + Manager` 的架构：`Worker` 部署在被监控节点上，定时采集 CPU、内存、磁盘、网络等核心运行指标，并通过 `gRPC + Protobuf` 统一上报到中心端；`Manager` 负责接收监控数据、进行基础评分与统计分析，并落库到 MySQL 供后续查询使用。相较于依赖内核模块或重量级监控框架的方案，该版本聚焦纯用户态实现，基于 Linux 的 `/proc` 伪文件系统与系统接口完成主要指标采集，在保证功能完整性的同时降低了实现复杂度，更适合作为可落地、可扩展的 C++ 系统项目。

## 主要内容

1. 设计并实现了基于 `Agent-Server` 的监控架构，将系统划分为数据采集端 `Worker` 和中心管理端 `Manager` 两部分；其中 `Worker` 负责本机指标采集与周期性推送，`Manager` 负责统一接收、解析、存储与查询，形成了从“采集—传输—存储—分析”的完整闭环。
2. 在数据采集层采用纯用户态方案，重点基于 Linux `/proc` 与标准系统接口完成多维度指标获取：通过解析 `/proc/stat` 与 `/proc/loadavg` 统计 CPU 使用率和负载信息，通过 `/proc/meminfo` 获取内存总量、空闲内存、可用内存与缓存等指标，通过 `/proc/net/dev` 结合两次采样差值计算网卡收发速率与包速率，通过 `/proc/diskstats` 统计磁盘吞吐、IOPS、时延和利用率，并结合主机名、IP 地址等信息完成节点标识。
3. 使用 `Protobuf` 统一定义监控消息结构，将 CPU、内存、磁盘、网络及主机基础信息封装为统一的 `MonitorInfo` 数据对象；基于 `gRPC` 构建 `Worker -> Manager` 的长连接通信链路，实现多节点监控数据的周期性上报，降低手写序列化和网络通信代码带来的复杂度，并提升数据传输的一致性与可维护性。
4. 在服务端实现监控数据接收、主机管理、基础评分与 MySQL 持久化逻辑：收到上报数据后，按主机维度维护最新状态，对 CPU、内存、磁盘、网络等关键指标进行综合评分，并将时间序列数据写入 MySQL；同时封装基础查询接口，支持按主机、时间范围查询监控记录，并进行简单趋势统计与状态分析。
5. 为了验证监控系统的有效性，结合 `stress`、`iperf3` 等工具构造 CPU、内存和网络压力场景，对比压测前后的指标变化，验证采集逻辑与上报链路的正确性；同时通过 `Docker` 对 `Manager`、MySQL 等组件进行容器化部署，降低环境搭建成本，为后续扩展告警、eBPF 网络观测或可视化展示预留空间。

## 个人收获

1. 熟悉了 Linux 系统常见监控指标的来源与计算方式，能够从 `/proc` 伪文件系统中提取原始数据，并通过差分采样计算 CPU 使用率、网卡吞吐、磁盘 IOPS 等动态指标；对“用户态获取系统运行状态”的实现方式有了较系统的理解。
2. 对 `gRPC + Protobuf` 的工程化使用有了较完整的实践经验，掌握了从协议定义、代码生成、消息封装到客户端上报、服务端接收处理的完整流程，加深了对 RPC 通信、二进制序列化以及服务拆分思路的理解。
3. 熟悉了 C++ 在系统项目中的模块化组织方式，能够围绕“监控器抽象—采集调度—网络传输—服务端处理”拆分代码结构，并在实际开发中使用 CMake 管理多模块工程，提升了对中等规模 C++ 项目的阅读、修改与重构能力。
4. 理解了监控系统不仅是“把数据采上来”，还需要考虑主机标识、评分模型、数据库表设计、时间序列查询、压测验证与部署方式等工程细节；对如何把一个单点采集程序逐步演化为可落地的完整项目有了更清晰的认识。
5. 在项目裁剪过程中强化了自己的工程取舍能力：针对当前阶段，优先保留 CPU、内存、磁盘、网络采集以及 gRPC、MySQL 等核心链路，先完成可运行、可讲解、可验证的基础版本，再为后续补充 eBPF、弱网分析、异常检测等高阶能力打下基础。
