构建一个**面向高并发Web场景**（比如选课系统、成绩查询系统）的**轻量级分布式性能监控与调度系统**。

机房里有 N 台部署了同一套 Web 应用的服务器，前端流量入口需要一个负载均衡器来分发请求。传统的负载均衡（比如 Nginx 轮询）是盲分，不考虑每台服务器的实时负载状态。我们的系统让每台服务器上的 **Worker 代理** 持续采集六大类指标，通过 gRPC 上报给 **Manager**，Manager 根据这些指标计算一个 **0-100 的综合健康评分**，再用**平滑加权轮询算法**选出当前最优的后端节点。

主要监控四类，cpu使用率/负载，内存，磁盘IO，网络收发速率。

## 对比XML JSON Protobuf

XML和JSON序列化后是一个文本语言，数据量大。

protobuf是二进制编码，数据量要比前两者都小，且序列化更快。json是运行时反射解析，天然比编译器生成的序列化代码慢。

JSON优点在于可读性，可以直接看。

protobuf支持跨语言编译，且扩展性好，每个字段都是数字编号，xml和json都是字段名序列化，修改起来更复杂。

# 数据采集

## 采集方式

/proc采低频数据，比如内存，负载，直接读/proc文件，解析文本就可以。

内核模块mmap采集高频数据，比如cpu使用情况。因为mmap避免了内核态到用户态的数据拷贝过程，开销更低。

ebpf采集网络数据。ebpf运行中内核的受限虚拟机中，挂载监控到内核的网络协议栈上。通过ebpf map与用户空间交互获取数据。

​	优点：ebpf可以在系统运行时随时加载程序，像插件一样扩展内核功能。

​		运行在虚拟机中，安全性好，适合网络流量的实时监控，性能开销很低。

## 具体实现

### cpu负载 /proc

从/proc/loadavg中读取一行数据，取出前三个字段作为过去1/5/15分钟的平均负载值。

平均负载值就是指单位时间可运行状态(正在用cpu或排队等cpu)和不可中断睡眠状态(等待IO)的进程数平均值。理想值等于cpu个数，超过说明有过载。

根据历史数据判断负载的变化趋势。

数据很小只有一行，且不是瞬时变量，所以用/proc。

### cpu利用率

统计的是cpu的非空闲状态时间比，也就是（总时间-空闲时间）/总时间。

平均负载多统计了不可中断睡眠状态，一般是等待IO的进程。

所以如果负载高但是cpu利用率低，说明可能是IO出现瓶颈，进程在等文件读写。

cpu利用率和负载都高才说明进程在大量计算，cpu处在瓶颈，需要减少并发或加cpu核心。

利用率高，负载低则是理想状态。

### cpu状态采集

内核模块优点：性能更好，采集高频数据时用mmap进行映射后不需要系统调用，开销极低。

编写内核模块，通过定时器操作每秒遍历所以cpu，读取kcpustat数组中的数据，统计自系统启动以来的每个cpu的累积数据，填写到共享区域，并注册字符设备。用户空间可以通过mmap把该共享内存映射的自己的地址空间。

用户空间中打开/dev字符设备，通过mmap映射共享内存，收集cpu数据，与上一次数据做差拿到真实数据。

## 数据在内核存储位置

每个cpu都有一个kstat cpu的结构体，其中包含一个cpustat数组，用于记录cpu在不同状态下的累积数据。

-  `user`：用户态消耗的 CPU 时间，比如普通应用程序的运行时间。
- `system`：内核态消耗的 CPU 时间，比如系统调用、内核服务等。
- `idle`：CPU 空闲时间，没有任务需要处理时的时间。它不包括等待 I/O 的时间（iowait)。
- `nice`：以“低优先级”运行的用户进程消耗的 CPU 时间。也就是进程的 nice 值被调整为 1-19 之间时的 CPU 时间。nice 可取值范围是 -20 到 19，数值越大，优先级反而越低。CPU 调度器会减少其时间片分配。
- `io_wait`：CPU 等待 I/O 操作（如磁盘、网络）完成的时间。
- `irq`：CPU 处理硬件中断（如网卡、磁盘等设备中断）的时间。
- `soft_irq`：CPU 处理软中断（如网络包处理、定时器等异步任务）的时间

### 指标意义

- 通过分析 user、system等比例，区分是应用层压力大（比如业务进程消耗多），还是内核态消耗高（比如系统调用、内核服务占用多）
- io_wait、irq、soft_irq 这些指标可以帮助我们定位更具体的性能问题， 比如 io_wait 很高，说明 CPU 大量时间在等磁盘或网络 I/O，可能是存储或网络成为瓶颈；irq 或 soft_irq 很高，可能是中断特别多、网络压力大或者驱动异常导致的。

### 排查

cpu满载：1.业务本身计算量大，用户态占用大。2.内核态消耗高，比如系统调用频繁，有大量内核模块操作。3.中断压力大，比如网络流量大，磁盘io密集。

cpu负载高cpu利用率很低：磁盘或网络太慢，导致很多程序都在等io，实际上没怎么用cpu

## 内存信息

从/proc/meminfo中读取文件内容。内存使用率：(总内存 - 可用内存)/总内存

### 主要内容

系统总内存大小(KB) 未使用内存(KB) 可用内存(包含未使用内存和可回收缓存)

可回收缓存：buffers块设备缓存：对磁盘数据的缓存，优化磁盘写操作。cached：文件内容缓存，优化读操作。

swapcached：在swap分区中缓存的内存。swap是用磁盘空间换取更少的IO操作，当系统内存紧张时，内核会把部分内存页放到磁盘的swap分区，内核会对这些数据进行标记，以后如果内存又不够，就可以直接丢弃内存中的页面，不用像之前一样再做一次换出。

`dirty`等待写入磁盘的脏页数 (KB)、`writeback`正在写回磁盘的内存页 (KB)

anonpages：匿名页面的内存使用总量。包含了进程的堆栈，bss段。

进程中的代码段，data段，共享库，则放在硬盘文件页。

选择proc原因：

* 内存相关数据在内核中分布广，用内核模块方式复杂
* 内存信息更新频率低，内核模块更适合高频数据的采集

### 排查

首先看系统使用的总内存，可用内存，缓存，判断整体的内存使用情况。

swap缓存过高，说明系统正在频繁吧内存页交换到硬盘，表示系统有内存压力，物理内存不足。

脏页（Dirty）、写回页（Writeback）来分析 磁盘I/O 压力，如果 `Dirty` ` writeback`持续高值，表示大量内存页等待写入磁盘，磁盘处理不过来 ，磁盘I/O 性能瓶颈，脏页积压，导致系统卡顿。

匿名页面的内存可以看应用层的内存消耗，主要是应用程序堆栈，malloc等动态分配的内存，如果持续增长说明程序在不断申请匿名内存但没有释放，可能有内存泄漏。

## ebpf

挂载点：

* XDP：挂载在网卡驱动的最早入口，延迟低，只统计数据流入的方向
* TC：挂载在[协议栈](内核里完整的网络收发代码（IP 路由、TCP 状态机、拥塞控制、分片重组等）)入口，可以双向统计

选择tc：可以分别统计入方向和出方向的流量，适合需要统计收发流量的场景。

统计量：统计每个网卡收发多少字节和数据包，具体就是接收/发送速率和接收包/发送包速率。

通过__atomic_fetch_add保证原子性。

ebpf程序：加载到内核后，会在网络包到达时触发，统计相应指标，包括收发字节数，包数，然后把这些数据存到ebpf的哈希表中，这个表是内核与用户空间共享的哈希表。用户通过结合上一次采集的结果和时间点，计算每个网卡的流量速率和包速率。

哈希表：因为每个网卡都需要独立的存储，网卡名是独立不连续的key，适合哈希查找。

### 选择原因

ebpf监控网络：网络流量是一个高频事件，每秒都可能有大量数据包通过不同的网卡进行传输，我们需要对这些数据进行实时统计，/proc有一定的延迟。而对应这种发生在内核网络路径上的动态事件，eBPF 可以直接挂在 `tc ingress/egress` 这些内核 hook 上，在包经过时就地统计，路径更短、时效性更好、扩展性也更强。

为什么不用mmap：内核中虽然也维护了网络统计，但它是分散的，不像cpu那样有一个kcpustat的全局变量，统计起来不方便。

其次内核对cpu的全局变量的更新是通过定时器触发，所以可以用mmap加一个定时器来读，而网络中，数据包随时可能到达，这时用ebpf在协议栈挂一个hook来统计数据更好。

最后ebpf程序在加载前会经过bpf验证器，这使得运行在内核里的ebpf程序即使有bug也不会拖垮整个内核。

排查：流量速率变化明显，那边有出现网络问题。

# 系统评分

计算变化率（全局保留每台主机的上一次采样）

评分是一个加权反向归一化的复合评分，cpu，内存，负载，磁盘，网络赋予不同权重

加权得到最终分。

### manager工作流

worker通过grpc 的SetMonitorInfo推送，manager接收到后计算变化率和分数，将算好的结果异步入队，线程消费队列，写入mysql。

分数有两个消费方，一个是基于分数做平滑加权轮询，分高的机器给更多流量。另外一个是历史查询，返回完整数据。

平滑轮询：最初是多个 Worker 上报性能数据，Manager 给它们打分，然后根据分数高低把请求调度到最健康的机器上。

但是在高并发情况下会出现评分最高的机器被连续分配过多请求导致性能严重下降，而且下一轮评分出来后，A 分数降了，流量又全部切到 B。

所以通过学习nginx添加了平滑加权轮询，每次最高分机器被选中后会减去一个权重，使得可以把流量分给其他机器，这样就能把连续调度打散，避免短时间内反复请求同一台机器，从而消除出现的问题。
